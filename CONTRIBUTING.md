# 🤝 Руководство по вкладу в проект

Спасибо за интерес к этому проекту! Если вы хотите внести свой вклад, пожалуйста, прочитайте это руководство.

## 📋 Требования к вкладу

### Перед началом

1. Сделайте **Fork** репозитория
2. Создайте **новую ветвь** для вашей функции:
   ```bash
   git checkout -b feature/description-of-feature
   ```
3. Убедитесь, что код компилируется без ошибок
4. Добавьте комментарии к сложным логикам
5. Протестируйте ваши изменения

### Стиль кода

Следуйте существующему стилю проекта:

```cpp
// Используйте PascalCase для имен классов
class TwoPhaseSimpleMergeSort { }

// Используйте camelCase для переменных и методов
void myMethod() { }
int myVariable = 0;

// Используйте kebab-case для имен файлов
// (уже установлено: FileGenerator.h, SortStats.h)

// Добавляйте пробелы вокруг операторов
int result = a + b;  // ✅
int result=a+b;      // ❌

// Используйте явные типы для шаблонов
TwoPhaseSimpleMergeSort<int> sort(filename, &stats);

// Документируйте сложные функции
// Пример:
// Читает серию из файла, начиная с позиции offset
// Возвращает количество прочитанных элементов
int readSeries(ifstream& file, int offset) { }
```

### Соглашения об именах

| Объект | Соглашение | Пример |
|--------|-----------|---------|
| Классы | PascalCase | `TwoPhaseSimpleMergeSort` |
| Методы | camelCase | `simpleMergeSort()` |
| Переменные | camelCase | `sortStats`, `fileSize` |
| Константы | k + PascalCase | `kTemp1`, `kMaxSize` |
| Файлы | PascalCase | `FileGenerator.h` |

## 🐛 Отправка отчета об ошибке

### Структура отчета

**Название**: Кратко опишите ошибку
```
[Баг] Падение программы при сортировке пустого файла
```

**Описание**:
```
Шаги для воспроизведения:
1. Запустить программу
2. Выбрать опцию "Создать входной файл"
3. Ввести 0 элементов
4. Выбрать любой алгоритм сортировки
5. Программа вызывает исключение

Ожидаемое поведение:
- Вывести сообщение об ошибке
- Вернуться в меню

Актуальное поведение:
- Программа падает с ошибкой доступа к памяти

Окружение:
- ОС: Windows 10
- Компилятор: MSVC (Visual Studio 2022)
- C++ Standard: C++17
```

## ✨ Предложение новой функции

### Структура предложения

**Название**: Краткое описание функции
```
[Функция] Экспорт результатов в CSV
```

**Описание**:
```
Описание проблемы:
Сейчас результаты доступны только в консоли, что затрудняет анализ 
и сравнение множества запусков.

Предлагаемое решение:
Добавить опцию экспорта результатов сортировки и статистики в CSV файл
для дальнейшего анализа в Excel или других инструментах.

Дополнительный контекст:
- Формат: filename,algorithm,cmp_count,move_count,time_ms
- Добавление к существующему файлу (append mode)
```

## 🔄 Pull Request процесс

### Шаг 1: Подготовка PR

```bash
# Обновите вашу ветвь с последними изменениями
git fetch origin
git rebase origin/main

# Или используйте merge (предпочтительно для длинных веток)
git merge origin/main
```

### Шаг 2: Создайте PR

На GitHub нажмите **New Pull Request** и заполните форму:

```markdown
# Описание

Краткое описание изменений

Fixes #(issue)  # Если закрывает issue

## Тип изменений

- [ ] Баг-фикс (изменения, которые исправляют проблему)
- [ ] Новая функция (изменения, которые добавляют функциональность)
- [ ] Breaking change (изменения, которые нарушают существующую функциональность)
- [ ] Обновление документации

## Как это было протестировано?

Опишите тесты, которые вы запустили:
- Тест A
- Тест B

## Чек-лист:

- [ ] Код следует стилю этого проекта
- [ ] Я провел самопроверку моего собственного кода
- [ ] Я добавил комментарии, особенно для сложных областей
- [ ] Я обновил документацию (если необходимо)
- [ ] Мои изменения не генерируют новые предупреждения
- [ ] Код компилируется без ошибок
- [ ] Я протестировал функциональность
```

### Шаг 3: Процесс рецензирования

- Автор проекта проверит ваш код
- Могут быть запросены изменения
- Обновляйте PR ветвь при необходимости:
  ```bash
  git add .
  git commit -m "Обновление согласно отзывам рецензентов"
  git push origin feature/your-feature
  ```

### Шаг 4: Слияние

После одобрения ваш PR будет объединен в `main` ветвь.

## 📝 Коммит сообщения

Используйте ясные и описательные коммит сообщения:

```bash
# ✅ Хорошо
git commit -m "Добавить проверку на пустой файл в TwoPhaseSort"
git commit -m "Оптимизировать операции чтения файла с буферизацией"
git commit -m "Исправить утечку памяти в слиянии серий"

# ❌ Плохо
git commit -m "Исправить баг"
git commit -m "WIP"
git commit -m "test123"
```

### Формат коммит сообщения

```
<тип>(<область>): <краткое описание>

<дополнительное описание>

<ссылка на issue>
```

**Типы**:
- `feat` - новая функция
- `fix` - исправление ошибки
- `docs` - изменения документации
- `refactor` - переработка кода без изменения функциональности
- `perf` - улучшение производительности
- `test` - добавление тестов
- `build` - изменения в системе сборки

**Области**:
- `TwoPhaseSort` - двухфазная сортировка
- `OnePhaseSort` - однофазная сортировка
- `FileGenerator` - генератор файлов
- `UI` - пользовательский интерфейс
- `docs` - документация

**Примеры**:
```
feat(TwoPhaseSort): добавить поддержку типа double
fix(OnePhaseSort): исправить ошибку при слиянии серий
docs(README): обновить примеры использования
perf(FileIO): оптимизировать операции чтения файла
```

## 🧪 Тестирование

### Основные сценарии тестирования

```cpp
// 1. Пустой файл
// 2. Один элемент
// 3. Два элемента
// 4. Отсортированный массив
// 5. Обратно отсортированный массив
// 6. Случайный массив
// 7. Массив с дубликатами
// 8. Большой массив (>1000000 элементов)
```

### Запуск программы в тестовом режиме

```bash
# Компиляция в режиме отладки
cl /std:c++17 /Zi main.cpp /o debug_main.exe

# Запуск с вводом из файла
echo 1 > input.txt
echo 100 >> input.txt
# (затем запустите программу)
```
